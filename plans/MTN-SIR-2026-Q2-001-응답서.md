## 1. 핵심 아키텍처 결정 — 시스템을 부풀리지 말고 LLM 위원회 프롬프트를 보강하라

위원회의 다수 지적은 **MTN 정량 엔진의 구조적 한계**(가격 기반 SEPA, 펀더멘털 미반영, Moat·회계 품질 자동 산출 불가)에서 기인한다. 이 한계는 시스템 명칭 변경(예: "SEPA 9/9" → "Price-SEPA")이나 추가 점수 신설로 해결되지 않는다. 정확한 해법은 **LLM 위원회 호출 단계에서 갭을 메우는 것**이며, 이는 MTN의 원래 설계 의도다.

## 2. §A 즉시 수용 — 데이터·산출 결함 (P0)

위원회의 8개 지적은 **데이터 무결성 또는 산출 로직 정합성** 문제이며, 모두 다음 sprint에서 처리한다.

### A-1. `avg_dollar_volume` 단위 오류 [James 지적]

GEV $270,920,000,000은 시가총액 수준값이며 일평균 거래대금이 될 수 없음. Yahoo 어댑터(`lib/finance/providers/yahoo-api.ts`) 단위 변환 또는 누적값을 평균으로 잘못 라벨링한 경로 추적.
- **수정 방향**: 일평균 거래대금 = (직전 20거래일 종가 × 거래량)의 산술평균. 단위는 USD.
- **검증**: 후보 10개의 ADV가 시가총액 대비 0.1~5% 범위에 들어오는지 sanity check.

### A-2. `vcp_status: 'none'` 일관성 오류 [Alex 지적]

VCP 점수 14~24점인 종목 전체에 `status='none'`이 표시됨. 점수와 status 분류 임계값이 불일치.
- **수정 위치**: `lib/master-filter/compute.ts` VCP 채점·라벨링 함수
- **방향**: status 매핑을 점수 기반 단계함수로 명시 (예: 0~10=none, 11~20=forming, 21~25=tight). 또는 status 분류 로직 자체를 재구현.

### A-3. `recommendation_tier: 'Low Priority'` 모든 후보 동일 [James 지적]

1위~10위 모두 동일 라벨. 분류 임계값이 미작동.
- **방향**: 분포 백분위 기준으로 재라벨 (상위 20% Recommended, 20~50% Partial, 그 외 Low).

### A-4. `mtn_recommendation: 'WATCH'` 모든 후보 동일 [Michael 지적]

1위~10위 모두 WATCH. 강도 차등 부재.
- **방향**: 점수 + 신뢰도 결합 임계값으로 BUY/WATCH/SKIP 분류. baseDays<5는 자동으로 WATCH 상한.

### A-5. null 필드 — 근본 원인 추적·버그 수정 [Alex 지적]

**UI 숨김 처리는 금지**. null이 노출된 모든 필드(`rs_rating`, `rs_percentile`, `ibd_proxy_score`, `mansfield_rs_flag`, `base_type`, `macro_action_level` 등)에 대해 다음을 수행:
1. null이 산출된 호출 경로를 코드 레벨에서 추적
2. **파이프라인 버그(데이터 어댑터·파서·단위 변환·컬럼 매핑·직렬화)** 가 원인이면 모두 수정
3. 정당한 데이터 부재(예: 신생 종목 1년 미만)만 명시적 결측 라벨 유지
4. 결과: 컨테스트 응답 페이로드의 null 필드는 "정당한 결측" 또는 "0"이 되도록

별도 체크리스트 문서: `MTN-P0-bug-hunting-checklist.md`

### A-6. `baseDays<5` 시 `stopPlan` 자동 출력 차단 [Michael 지적]

baseDays=1 종목(COHR, CIEN, DELL)에 stopPrice가 친절하게 출력되어 사용자에게 잘못된 안전감 제공.
- **방향**: `baseDays<5`이면 `stopPlan = null` + 명시적 플래그 `INSUFFICIENT_BASE_FOR_STOP_REFERENCE: true`. UI는 이 플래그를 받아 손절 표시 자체를 숨기고 "Base 미형성으로 손절 기준선 산출 보류" 문구 출력.

### A-7. 신뢰도 0.61~0.63 출력에 디스클레이머 강제 [David 지적]

신뢰도 분포가 좁게 갇힌 상태(데이터 결함의 결과). UI 카피 추가:
> "본 신뢰도는 정량 점수의 자체 일관성 지표이며 펀더멘털·매크로 검증을 대체하지 않습니다. 외부 LLM 위원회 평가와 병행하십시오."

### A-8. P3 Macro Regime 100점 만점 산출 로직 감사 [Michael 지적의 재해석]

위원회는 "만점 = 정점 신호"로 해석했으나, 그 지적의 더 본질적 함의는 **만점이 실제 지표로부터 산출된 정상값인가**를 검증해야 한다는 것.
- **감사 대상**:
  - `lib/master-filter/compute.ts` Macro Regime 산출식
  - 입력 변수: M1(FRED), 달러지수(FRED), 구리/금 비율(Yahoo), 분산일(자체 계산), FTD(자체 계산)
  - 각 입력의 최신 데이터 fetch 시점, 결측 시 fallback 동작
- **결함 유형 검증**:
  - 임의 보정·기본값 의존이 발견되면 → P0 버그로 확정 처리
  - 모든 입력이 정상 fetch되어 만점 산출이 정당하면 → 사용자 해석 가이드를 Glossary `Regime Score`에 보강

별도 체크리스트 문서: `MTN-P0-bug-hunting-checklist.md`

## 3. §A-bis RS 데이터 무조건 복구 — P0 격상 [Alex 지적]

위원회 회의록 기준 부분 수용 항목이었으나, **컨테스트에서 RS가 null로 노출되는 것 자체가 버그**다. 디스클레이머로 회피하지 않고 출력 복구.

- **현황**: `lib/finance/market/rs-proxy.ts`에 IBD Proxy + Mansfield RS + Universe-relative percentile이 이미 구현되어 있음
- **버그 가설**: 분석 → 후보 추출 → 컨테스트 페이로드 직렬화 단계 중 어디선가 끊김
- **수정 절차**:
  1. RS Proxy 함수 단위 호출에서는 정상 산출되는지 단독 테스트
  2. 후보 추출 단계(`lib/master-filter/...`)에서 RS 필드가 채워져서 전달되는지 확인
  3. 컨테스트 페이로드 직렬화(`app/api/contest/sessions/[id]/analyze/route.ts`)에서 RS 필드가 누락·덮어쓰기되는지 확인
  4. 파이프라인 어느 단계에서든 RS가 살아남도록 수정

## 5. §C 거부 — 시스템 정체성·인프라 한계 위반 (P4)

다음 6개 항목은 명시적으로 시행하지 않는다. 거부 근거를 회의록에 기록.

### C-22. 임베딩 기반 동적 클러스터링 [David 지적, 거부]

후보 10개에 임베딩 + 상관행렬 동적 클러스터링은 over-engineering. GICS sub-industry로 95% 효과(B-12에서 채택). 추가 정확도 향상 < 구현·유지보수 비용.

### C-23. Fama-French 5-factor / Barra 팩터 노출도 [Alex 지적, 거부]

Barra는 유료 라이선스 필수. F-F 5팩터 자체 회귀는 가능하나 **추세매매 시스템에서 momentum factor 노출이 양(+)인 것은 자명**하므로 산출이 의사결정에 추가하는 정보가 미미. 정보 가치 < 구현 비용.

### C-24. Regime classifier별 신뢰도 차등 [Alex 지적, 거부]

Master Filter + Macro Regime이 이미 regime classifier 역할. 별도 mean-reversion / momentum regime 분류기는 동일 정보의 재포장. 정보 중복.

### C-25. Monte Carlo 포트폴리오 손실 시나리오 [Michael 지적, 거부]

후보 10개·과거 1년 데이터 기준으로 추정한 공분산 행렬은 **통계적으로 매우 불안정**. Monte Carlo 출력은 spurious precision(가짜 정밀도)을 사용자에게 노출하는 위험이 산출 가치보다 큼. **B-12의 GICS 클러스터 경고로 80% 동일 효과**.

### C-26. Almgren-Chriss 체결 시뮬레이션 [James 지적, 거부]

시장 충격 모델은 hedge fund 트레이더 도구. MTN은 개인용 추세매매 보조이며, 사용자 평균 주문 수량은 ADV의 0.01% 이하로 추정됨. 스프레드·슬리피지 우려는 B-15의 `LIMIT_ORDER_RECOMMENDED` 플래그로 충분.

### C-27. 가중치 백테스트 기반 재검증 [Alex 지적, 부분 거부]

재검증 자체는 옳으나 **백테스트 인프라(D-20)가 선결**. 가중치 자체(VCP25/RS25/SEPA20/Mom15/Tech15)는 Minervini/O'Neil 문헌과 일관됨. 백테스트 인프라 갖춘 뒤 분기별로 재교정.

## 8. 위원회에 대한 참고

본 응답서는 회의록 33개 지적 중 **수용 가능한 것은 전부 수용**하고, 거부 항목에 대해서는 **근거를 명문화**했다. 거부 근거는 (a) 무료 인프라 제약, (b) 시스템 정체성(셋업 스크리너이지 OEMS·Bloomberg 단말이 아님), (c) 통계적 spurious precision 위험, (d) 정보 가치 < 구현 비용 — 이 4가지로 분류된다.

위원회의 정성적 판단은 LLM 프롬프트 보강(§1)을 통해 시스템 출력 단계에서 직접 결합되도록 설계한다. 이는 시스템 결함이 아니라 **MTN의 원래 설계 의도(정량 엔진 + LLM 위원회 결합)** 가 실현되는 방향이다.
