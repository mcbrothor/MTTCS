# SKILL: 병렬 태스크 분해

## 로드 시점
서브태스크가 3개 이상이거나, 실행 순서가 불명확할 때.

---

## 분해 절차

1. 전체 서브태스크를 순서 없이 나열
2. 각 쌍 (A, B): B가 A의 출력을 필요로 하면 A → B 엣지 추가
3. 들어오는 엣지가 없는 태스크들 → Wave 1
4. Wave 1 제거 후 반복 → Wave 2, Wave 3...
5. 같은 Wave 내 태스크는 전부 병렬 실행

---

## 의존성 판단 기준

| 질문 | Yes | No |
|------|-----|----|
| B가 A가 쓴 파일을 읽는가? | 순차 | 병렬 |
| B가 A의 심볼을 import하는가? | 순차 | 병렬 |
| B가 A의 반환값을 사용하는가? | 순차 | 병렬 |
| A와 B가 동일 파일을 수정하는가? | 순차 | 병렬 |
| A와 B가 가변 상태를 공유하는가? | 순차 | 병렬 |

위 조건 중 해당 없음 → **병렬이 기본값**.

---

## 작업 유형별 Wave 패턴

### 신규 기능 (Greenfield)
```
Wave 1: 데이터 모델 | config/env 구조 | 타입 정의
Wave 2: 비즈니스 로직 | 데이터 접근 계층 | 외부 서비스 클라이언트
Wave 3: API handler | 백그라운드 작업
Wave 4: 통합 테스트 | 문서 업데이트
```

### 버그 수정
```
Wave 1: 실패 테스트 작성 (재현) | 영향 파일 전체 읽기
Wave 2: 수정 구현
Wave 3: 수정 테스트 통과 확인 | 회귀 테스트 실행
```

### 리팩토링
```
Wave 1: 모든 call site 파악 | characterization 테스트 작성
Wave 2: 구조 변경 (테스트 green 유지)
Wave 3: dead code 제거 | call site 업데이트 | 문서 업데이트
```

### 리서치 + 구현
```
Wave 1: 외부 문서/소스 읽기 | 기존 코드베이스 읽기
Wave 2: 구현 (Wave 1 결과 반영)
Wave 3: 테스트 및 검증
```

---

## 피해야 할 패턴

**가짜 병렬화** — 하나의 논리적 단위를 즉시 서로 의존하는 두 조각으로 쪼개는 것.
신호: Wave 2가 Wave 1의 부분적 결과 없이는 시작 불가.

**과도한 순차화** — 서로 import 관계가 없는 파일 A, B, C를 하나씩 순서대로 작성.
독립 파일은 동시에 작성한다.

**조기 통합** — Checkpoint 통과 전에 모듈 통합 시도.
Wave N 검증 완료 후에만 Wave N+1을 시작한다.
