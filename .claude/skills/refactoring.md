# SKILL: 리팩토링

## 로드 시점
관찰 가능한 동작을 바꾸지 않으면서 기존 코드 구조를 변경할 때.

---

## 사전 조건 (건너뛰지 않음)

코드를 건드리기 전에:

1. **테스트가 존재하고 통과한다.** 없으면 characterization 테스트를 먼저 작성한다 — 현재 동작을 문서화하는 테스트. 동작이 잘못됐더라도 안전망이 필요하다.
2. **범위가 정의되어 있다.** "전체 리팩토링"은 태스크가 아니다. 어느 모듈, 어느 추상화, 어느 인터페이스인지 경계를 정의한다.
3. **목표가 명시되어 있다.** 가독성, 성능, 확장성 리팩토링은 각각 다른 변환이 필요하다. 무엇을 하는지 알고 시작한다.

---

## 실행 규칙

- **한 번에 하나의 변환.** 이름 변경, 그다음 추출, 그다음 이동.
  동시에 하면 diff가 읽기 불가능하고 버그 위치 파악이 어렵다.
- **모든 변환 후 테스트가 green 상태를 유지한다.** green 상태에서 커밋.
  테스트가 깨지면 마지막 green 상태로 revert한다.
- **리팩토링 중에 버그를 수정하지 않는다.** 발견하면 메모하고 별도 커밋으로 처리한다.
  동작 변경과 구조 변경을 섞으면 코드 리뷰가 불가능하다.

## 변환 카탈로그

### Extract Function
- **시점**: 코드 블록에 무엇을 하는지 설명하는 주석이 필요할 때
- **신호**: `# 할인 계산` 주석을 달려고 할 때
- **방법**: 블록을 이름 있는 함수로 추출. 함수명이 주석을 대체한다.

### Inline Variable/Function
- **시점**: 추상화가 명확성 없이 간접 참조만 추가할 때
- **신호**: 함수명이 one-liner가 하는 일과 동일할 때

### Replace Conditional with Polymorphism
- **시점**: `if type == X` 패턴이 여러 곳에 반복될 때
- **방법**: base class / protocol 정의, 동작을 구현체로 이동

### Introduce Parameter Object
- **시점**: 함수가 관련된 인자를 4개 이상 받을 때
- **방법**: dataclass/struct로 그룹화. 인자 1개, 명명된 필드들.

### Decompose Large Module
- **시점**: 모듈의 역할을 "and" 없이 설명하기 어려울 때
- **방법**: 줄 수가 아닌 책임 기준으로 분리

## 안전한 Rename 순서

대규모 코드베이스에서 이름 변경 시:

1. 새 이름을 alias로 추가: `NewName = OldName`
2. 모든 call site를 새 이름으로 마이그레이션 (가능한 자동화)
3. 모든 call site 업데이트 후 alias 제거
4. 기존 이름 제거

1단계와 4단계를 같은 커밋에 넣지 않는다.

---

## 완료 기준

리팩토링 완료 조건:
- 전체 테스트 통과 (기존 suite + 추가된 characterization 테스트)
- diff가 읽기 가능 — 리뷰어가 각 변경을 독립적으로 이해 가능
- 동작 변경 없음 (검사가 아닌 테스트 suite로 검증)
- dead code는 주석 처리가 아닌 삭제
