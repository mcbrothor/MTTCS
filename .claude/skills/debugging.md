# SKILL: 디버깅 & Root Cause 분석

## 로드 시점
버그 원인이 즉시 명확하지 않을 때. 증상이 원인을 직접 가리키지 않을 때.

---

## 디버깅 프로토콜

### Step 1: 코드 건드리기 전에 특성 파악
수정 전에 이 질문들에 답한다:

- 정확히 관찰되는 동작은? ("동작 안 함"은 답이 아님)
- 정확히 기대하는 동작은?
- 재현 가능한가? 항상 / 간헐적 / 특정 조건에서?
- 마지막으로 동작했을 때는? 그 이후 무엇이 바뀌었나?

### Step 2: 테스트로 재현
실패를 재현하는 가장 작은 테스트를 작성한다.
재현이 안 되면 아직 이해하지 못한 것이다.

### Step 3: 문제 공간 이분 탐색
매 단계에서 용의자의 절반을 제거한다.

```
전체 시스템 실패
  → DB 없이도 실패하는가?      → I/O 격리
  → 하드코딩 입력에서도 실패?   → 데이터 격리
  → 모듈 단독으로도 실패?       → 모듈 격리
  → 특정 입력에서만 실패?       → 케이스 격리
```

### Step 4: 직접 원인 특정
잘못된 출력을 만드는 구체적인 라인.
"race condition이다"가 아니라 "47번 라인이 goroutine이 쓰기 전에 변수를 읽는다."

### Step 5: 근본 원인 특정
직접 원인이 존재하는 이유.
"변수가 stale했다"가 아니라 "호출자가 async 함수에서 동기적 동작을 가정했다."

증상을 패치하지 말고 근본 원인을 수정한다.

## 증상별 일반적 Root Cause

| 증상 | 일반적 root cause |
|------|-------------------|
| 로컬 성공, CI 실패 | env var 누락, timezone 차이, 경로 구분자 |
| 간헐적 실패 | Race condition, 미초기화 상태, 외부 의존성 불안정 |
| 1회 성공, 재시도 실패 | 공유 상태 변이, 실행 간 reset 누락 |
| Off-by-one | 경계 포함/미포함 혼동 |
| 예상치 못한 None/null | Guard 누락, API 응답 구조 오가정 |
| 점진적 성능 저하 | 메모리 누수, unbounded cache, connection pool 고갈 |
| DB 데이터 오염 | 트랜잭션 누락, isolation level 오설정, async 쓰기 race |

## 계측 전략

필요한 최소한의 계측만 추가한다.

1. 실패 지점에서 **정확한 값을 출력/로그** — 설명이 아닌 값
2. **경계에서 로그** — 들어오는 것, 나가는 것
3. 수정 후 **계측 제거** — debug 로그를 코드에 남기지 않는다

전체에 로그를 뿌리지 않는다. 타겟팅 > 장황함.

---

## 중단 기준

3회 시도 후 진전이 없으면:
1. 현재 가설을 명확하게 적는다
2. 가설을 확인하거나 부정할 증거를 찾는다
3. 증거를 찾은 뒤 수정한다

가설 없는 시도 반복은 컨텍스트와 시간을 낭비한다.
